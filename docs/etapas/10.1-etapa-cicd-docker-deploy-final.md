GradeFlow - Etapa 10.1 - CI/CD, Docker e Deploy

Objetivo
Automatizar build, testes, empacotamento e deploy do projeto, alem de disponibilizar o sistema publicamente para avaliacao e demonstracao.

Quando implementar
Depois que login e permissoes (Etapa 10) estiverem funcionando.
Nao faz sentido publicar um sistema sem controle minimo de acesso.

Tecnologias
CI/CD:
- GitHub Actions

Containerizacao:
- Docker
- Docker Compose

Deploy backend:
- Azure App Service
- Railway
- Render

Deploy frontend:
- Vercel
- Netlify
- Azure Static Web Apps

Banco de dados:
- Azure SQL
- SQL Server em container, no ambiente local

Docker
Tarefas:
- Criar Dockerfile para GradeFlow.Api
- Criar Dockerfile para GradeFlow.Web
- Criar docker-compose.yml contendo GradeFlow.Api, GradeFlow.Web e banco de dados
- Configurar variaveis de ambiente via .env e/ou configuracoes especificas do ambiente
- Garantir a aplicacao automatica das migrations durante o deploy ou startup da GradeFlow.Api
- Implementar endpoint de Health Check em /health
- Configurar Health Checks dos containers

CI (Integracao Continua)
Workflow: .github/workflows/ci.yml

Deve executar em todo push e pull request:
- Restore e build do backend (.NET)
- Execucao dos testes automatizados com dotnet test
- Build do frontend com npm ci e npm run build
- Falhar o pipeline caso qualquer teste ou build apresente erro

CD (Entrega Continua)
Workflow: .github/workflows/deploy.yml

Deve executar apenas na branch main, apos o CI ser concluido com sucesso:
- Build da imagem Docker da GradeFlow.Api
- Build da imagem Docker do GradeFlow.Web
- Publicacao das imagens no GitHub Container Registry, o GHCR
- Deploy automatico para o servico escolhido

Deploy publico
Tarefas:
- Publicar backend com HTTPS valido
- Publicar frontend apontando para a URL publica da Api
- Configurar variaveis de ambiente do ambiente de producao
- Ajustar CORS da Api para o dominio publico do frontend
- Adicionar link do ambiente publico no README

Criterios de conclusao
A etapa sera considerada concluida quando:
- Backend estiver executando via Docker
- Frontend estiver executando via Docker
- Banco de dados estiver configurado e acessivel nos ambientes local e de producao
- Pipeline de CI estiver executando build e testes automaticamente
- Pipeline de CD estiver realizando deploy automatico
- Ambiente estiver acessivel publicamente via HTTPS
- Endpoint /health estiver respondendo corretamente
- README estiver atualizado com as instrucoes de acesso ao ambiente publicado

Ponto de atencao
Nao versionar senhas, tokens, chaves de API ou connection strings reais no repositorio.
Utilizar:
- GitHub Actions Secrets
- Variaveis de ambiente
- Configuracoes especificas do servico de hospedagem

A exposicao de credenciais compromete a seguranca do projeto e do ambiente de producao.
Alem disso, sem um ambiente publicado, as funcionalidades implementadas nas etapas anteriores so poderao ser avaliadas por quem clonar e executar o projeto localmente.
