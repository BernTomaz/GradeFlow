# 10.1A — Integração Contínua

## Objetivo

Criar o pipeline de CI do GradeFlow para validar backend e frontend em todo push e pull request para a branch principal.

## Arquivo esperado

```text
.github/workflows/ci.yml
```

## Inspeção obrigatória

Antes de alterar arquivos, confirmar:

- branch principal;
- localização de `GradeFlow.slnx`;
- versão real do .NET;
- existência de `global.json`;
- localização do `package.json`;
- versão real do Node;
- scripts disponíveis no `package.json`;
- existência de `.github/workflows`;
- dependências dos testes;
- necessidade de configuração JWT;
- necessidade de banco real nos testes.

Depois, prosseguir imediatamente.

## Workflow

Executar em:

- push para `main`;
- pull request direcionado a `main`.

Configurar:

```yaml
permissions:
  contents: read
```

Adicionar `concurrency` para cancelar execuções antigas da mesma branch ou pull request.

## Backend

Executar na raiz:

```powershell
dotnet restore GradeFlow.slnx
dotnet build GradeFlow.slnx --no-restore -m:1
dotnet test GradeFlow.slnx --no-build -m:1
```

Não executar `dotnet clean`.

Os testes JWT já usam configuração em memória. Não adicionar segredo ao workflow sem necessidade comprovada.

Caso o startup exija uma chave, usar apenas variável descartável de teste:

```text
Jwt__Key
```

A chave:

- deve ter pelo menos 32 bytes;
- deve existir apenas no job;
- não deve ser GitHub Secret se não tiver valor operacional;
- nunca deve ser usada fora dos testes.

## Frontend

Executar em:

```text
src/GradeFlow.Web
```

Comandos:

```powershell
npm ci
npm run build
npm test -- --watch=false
```

Usar somente flags compatíveis com os scripts reais.

## Cache

Pode usar:

- cache NuGet do `setup-dotnet`;
- cache npm baseado em `package-lock.json`.

O cache não pode substituir restore ou `npm ci`.

## README

Adicionar seção curta contendo:

- o que o CI valida;
- quando é executado;
- onde visualizar no GitHub;
- badge do workflow, somente se nome e caminho estiverem corretos.

## Critérios de aceitação

- `.github/workflows/ci.yml` existe.
- Push e pull request executam o CI.
- Backend restaura, compila e testa.
- Frontend instala, compila e testa.
- Qualquer falha interrompe o workflow.
- Nenhum segredo operacional está versionado.
- O CI não depende de ambiente local.
- README foi atualizado.
- Após o push, o workflow deve ser confirmado no GitHub.

## Validação local

```powershell
dotnet restore GradeFlow.slnx
dotnet build GradeFlow.slnx --no-restore -m:1
dotnet test GradeFlow.slnx --no-build -m:1
```

```powershell
cd src/GradeFlow.Web
npm ci
npm run build
npm test -- --watch=false
```

## Commit sugerido

```text
ci: add backend and frontend validation workflow
```

## Dockerfile da API

Criar Dockerfile multi-stage para:

- restore;
- build;
- publish;
- runtime final;
- usuário não root, quando compatível;
- exposição da porta correta;
- health check compatível.

## Dockerfile do frontend

Criar Dockerfile multi-stage para:

- `npm ci`;
- build Angular;
- servidor web de produção, preferencialmente Nginx;
- fallback para rotas Angular;
- proxy ou configuração para API pública;
- health check do container.

## Docker Compose

Criar:

```text
docker-compose.yml
```

Serviços:

- `gradeflow-api`;
- `gradeflow-web`;
- `sqlserver`.

Configurar:

- rede interna;
- volume persistente do banco;
- variáveis de ambiente;
- dependências de inicialização;
- health checks;
- portas documentadas;
- `.env.example`.

## Regras

- SQL Server em container é para desenvolvimento e demonstração local.
- Produção deve preferir banco gerenciado.
- Não versionar `.env`.

- Não aplicar migrations automaticamente nesta subetapa sem a decisão da 10.1D.

## Critérios de aceitação

- `docker compose build` passa.
- `docker compose up` inicia os três serviços.
- frontend acessa backend.

# 10.1D — Estratégia de Migrations

## Objetivo

Definir uma estratégia previsível e segura para atualizar o banco em cada ambiente.

## Decisão obrigatória

Avaliar e documentar uma das estratégias:

1. comando manual controlado;
2. job de migration separado;
3. etapa específica antes do deploy;
4. migration no startup apenas em ambiente local.

## Recomendação

Para produção, preferir execução única e controlada antes da atualização da aplicação.

Evitar `Database.Migrate()` indiscriminado no startup de múltiplas instâncias.

## Implementação esperada

- comando reproduzível;
- geração de script idempotente;
- job ou etapa de migration;
- falha explícita em caso de migration inválida;
- documentação de rollback e backup;
- teste com banco vazio;
- teste com banco existente.

## Critérios de aceitação

- banco vazio pode ser criado.
- banco existente pode ser atualizado.
- migration não executa concorrentemente.
- falha impede deploy.
- estratégia está documentada.
- backup/rollback estão descritos, mesmo que manualmente.

## Commit sugerido

```text
chore(database): define controlled production migration strategy
```
