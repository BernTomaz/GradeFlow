# GradeFlow - Etapa 10.1 - CI/CD, Docker e Deploy

## Objetivo

Automatizar build, testes, empacotamento e deploy do projeto, além de disponibilizar o sistema publicamente para avaliação e demonstração.

---

## Quando implementar

Depois que login e permissões (Etapa 10) estiverem funcionando.

Não faz sentido publicar um sistema sem controle mínimo de acesso.

---

## Tecnologias

### CI/CD

* GitHub Actions

### Containerização

* Docker
* Docker Compose

### Deploy Backend

* Azure App Service
* Railway
* Render

### Deploy Frontend

* Vercel
* Netlify
* Azure Static Web Apps

### Banco de Dados

* Azure SQL
* SQL Server em container (ambiente local)

---

## Docker

### Tarefas

* Criar Dockerfile para `GradeFlow.Api`
* Criar Dockerfile para `GradeFlow.Web`
* Criar `docker-compose.yml` contendo `GradeFlow.Api`, `GradeFlow.Web` e banco de dados
* Configurar variáveis de ambiente via `.env` e/ou configurações específicas do ambiente
* Garantir a aplicação automática das migrations durante o deploy ou startup da `GradeFlow.Api`
* Implementar endpoint de Health Check (`/health`)
* Configurar Health Checks dos containers

---

## CI (Integração Contínua)

### Workflow

```txt
.github/workflows/ci.yml
```

### Deve executar em todo push e pull request

* Restore e build do backend (.NET)
* Execução dos testes automatizados (`dotnet test`)
* Build do frontend (`npm ci` e `npm run build`)
* Falhar o pipeline caso qualquer teste ou build apresente erro

---

## CD (Entrega Contínua)

### Workflow

```txt
.github/workflows/deploy.yml
```

### Deve executar apenas na branch `main`, após o CI ser concluído com sucesso

* Build da imagem Docker da `GradeFlow.Api`
* Build da imagem Docker do `GradeFlow.Web`
* Publicação das imagens no GitHub Container Registry (GHCR)
* Deploy automático para o serviço escolhido

---

## Deploy Público

### Tarefas

* Publicar backend com HTTPS válido
* Publicar frontend apontando para a URL pública da API
* Configurar variáveis de ambiente do ambiente de produção
* Ajustar CORS da API para o domínio público do frontend
* Adicionar link do ambiente público no README

---

## Critérios de Conclusão

A etapa será considerada concluída quando:

* Backend estiver executando via Docker
* Frontend estiver executando via Docker
* Banco de dados estiver configurado e acessível nos ambientes local e de produção
* Pipeline de CI estiver executando build e testes automaticamente
* Pipeline de CD estiver realizando deploy automático
* Ambiente estiver acessível publicamente via HTTPS
* Endpoint `/health` estiver respondendo corretamente
* README estiver atualizado com as instruções de acesso ao ambiente publicado

---

## Ponto de Atenção

Não versionar senhas, tokens, chaves de API ou connection strings reais no repositório.

Utilizar:

* GitHub Actions Secrets
* Variáveis de ambiente
* Configurações específicas do serviço de hospedagem

A exposição de credenciais compromete a segurança do projeto e do ambiente de produção.

Além disso, sem um ambiente publicado, as funcionalidades implementadas nas etapas anteriores só poderão ser avaliadas por quem clonar e executar o projeto localmente.
